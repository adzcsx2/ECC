# /ecc:plan-doc

为任务生成完整文档集，并在全部执行文档完成后强制复审产品需求对齐、鲁棒性与可执行性，发现确定问题即修订并重新审计。

---

## 功能

- 开始一个需要完整文档的功能或任务时使用
- 为长期维护的功能建立文档基线
- 多人协作项目中快速生成可交接的文档集

## 用法

```
/ecc:plan-doc <task-slug> [test]
```

- `<task-slug>`：任务名称（小写连字符格式）
- `[test]`：可选，加上后额外生成测试文档

示例：

```
/ecc:plan-doc user-auth
/ecc:plan-doc payment-refactor test
/ecc:plan-doc test add-search-api
```

## 生成内容

文档集生成到 `docs/plan/<task-slug>-YYYY-MM-DD/` 目录下：

| 文件                 | 说明                                       |
| -------------------- | ------------------------------------------ |
| `README.md`          | 任务概述、状态和快速导航                   |
| `00-执行文档.md`     | 进度指针、执行清单、需求追踪与最终审计记录 |
| `01-架构设计.md`     | 架构决策、边界与约束                       |
| `02-开发规范.md`     | 必须/禁止规则、模板与反模式                |
| `03-修复路线图.md`   | 阶段、里程碑、依赖与回滚                   |
| `04-测试计划.md`     | 测试策略（加 `test` 参数时生成）           |
| `05-测试用例清单.md` | 测试用例与回归矩阵（加 `test` 参数时生成） |

## 新增特性

- **模块化执行契约**：Stage 3 读取结构契约并确认计划，Stage 4 直接落盘；全部文档写完后，Stage 5 重新读取结构契约和质量门并独立审计
- **并行分组分析**：当任务数 ≥3 且为 git 仓库时，自动按文件依赖关系分组并行调度，减少串行等待时间
- **非阻塞模型建议**：默认使用当前模型继续生成；仅在用户确认时显式要求等待切换才暂停
- **自动目录复用**：检测已有文档目录后自动复用，避免重复生成覆盖已有进度
- **生成后质量门**：全部文档写完后重新从磁盘读取文档契约、产品文档和执行文档，做契约符合性、双向需求追踪、跨文档一致性与鲁棒性审计
- **自动修订闭环**：确定的文档缺陷会做最小修订并完整复审；涉及产品意图取舍的冲突转为 blocker，不替用户猜测

## 行为说明

- 优先使用 `/ecc:execute-doc` 执行生成的 `00-执行文档.md`，不可用时再回退到 `/ecc:plan`
- 若用户拒绝安装相关依赖，降级为纯文本恢复提示
- **执行清单面向低能力 AI**：`00-执行文档.md` 的每个 `P<N>.<M>` 步骤强制写清「读什么 / 改什么 / 怎么验证 / 完成判据」，行号只作提示并配符号锚点，避免弱模型照行号盲改或跑偏
- **生成后回读校验**：写完所有文档后会列目录确认每个文件确实落盘且非空，被格式化/同步钩子误删的文件会自动补写
- **先生成后审计**：确认后的 Generation Handoff 直接驱动落盘，不在 Stage 4 前重复读取契约；契约重读和符合性检查统一放在全部文件落盘后的 Stage 5
- **产品 ↔ 执行双向对齐**：每条产品需求必须映射到执行项与验收证据；每个代码变更项也必须能追溯到产品需求或明确批准的工程支撑项
- **鲁棒性复审**：覆盖失败路径、边界校验、权限安全、超时取消、重试幂等、并发一致性、迁移兼容、资源清理、可观测性与回滚；不适用项必须说明原因
- **修订边界**：可确定的问题直接修订生成文档；若产品文档冲突或修复会改变产品意图，则将执行状态设为 `blocked` 并请求产品决策

## 模块结构

| 文件 | 加载时机 | 职责 |
| --- | --- | --- |
| `commands/plan-doc.md` | 命令启动 | 参数、确认门、目录和断点编排 |
| `references/document-contract.md` | Stage 3 前读取、全部落盘后在 Stage 5 重读 | 固定输出结构、进度指针、原子执行项契约及生成后符合性审计 |
| `references/subagent-routing.md` | Stage 3 前 | 技术栈识别、并行组与不可委派职责 |
| `references/post-generation-quality-gate.md` | 初始准备、Stage 5 前重读 | 产品对齐、鲁棒性、修订与复审闭环 |
