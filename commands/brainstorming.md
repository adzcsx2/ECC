# Brainstorming:把想法逼成已批准的设计

把模糊想法通过协作对话变成清晰、已获批准的设计,再交给 `/ecc:plan` 出实现计划。

先理解项目上下文,然后一次一个问题地澄清想法;理解清楚后,呈现设计并获取用户批准。**设计未经批准,绝不实现。**

## 硬门禁(HARD-GATE)

在呈现设计并获得用户明确批准之前,**禁止**:

- 写任何代码
- 调用任何实现类 skill 或 agent(`tdd-workflow`、`build-fix`、`frontend-design` 等)
- 搭建脚手架、创建项目、执行任何实现动作

这适用于**所有**项目,无论看起来多简单。

## 反模式:"这个太简单,不需要设计"

每个项目都要走这个流程——todo list、单函数工具、配置改动,全部都要。越是"简单"的项目,未经检验的假设造成的返工越多。设计可以很短(真正简单的项目几句话即可),但**必须呈现、必须获批**。

## 执行流程

任务: $ARGUMENTS

按以下顺序完成,每一项都是独立任务:

1. **探索项目上下文** —— 看文件、文档、近期 commit,理解当前结构
2. **评估范围** —— 若请求涉及多个独立子系统,先标记并帮助分解成子项目,再就第一个子项目走流程
3. **逐个澄清问题** —— 一次只问一个,搞清目的 / 约束 / 成功标准
4. **提出 2-3 个方案** —— 带权衡,给出推荐并说明理由
5. **呈现设计** —— 按复杂度分节,每节单独确认
6. **写设计摘要** —— 形成一份简洁、可直接喂给 `/ecc:plan` 的需求摘要
7. **设计自审** —— 查占位符 / 矛盾 / 歧义 / 范围,就地修
8. **用户批准** —— 等待明确同意后才进入终态
9. **终态:调用 `/ecc:plan`** —— 见下文

## 探索需求

- 先看项目状态(文件、文档、近期 commit)
- 评估范围:请求若描述多个独立子系统(如"一个带聊天、文件存储、计费、分析的平台"),立即标记,先分解再细化;对范围合适的项目,一次问一个问题来澄清
- 多选题优先于开放式提问
- 聚焦理解:目的、约束、成功标准

## 探索方案

- 提出 2-3 个不同方案,附权衡
- 用对话方式呈现,给出你的推荐和理由
- 先讲推荐方案,解释为什么推荐它

## 呈现设计

- 理解清楚要做什么之后,呈现设计
- 每节按复杂度伸缩:简单的几句话,复杂的最多 200-300 字
- 每呈现一节,问用户这节对不对
- 覆盖:架构、组件、数据流、错误处理、测试
- 随时准备回头澄清

## 为隔离与清晰而设计

- 把系统拆成职责单一的小单元,通过定义良好的接口通信,可独立理解和测试
- 对每个单元能回答:它做什么、怎么用、依赖什么
- 别人能否不看内部实现就理解一个单元的职责?能否改内部而不破坏调用方?不能就说明边界要打磨

## 在既有代码库中工作

- 提改动前先探索现有结构,遵循既有模式
- 当现有代码的问题影响当前工作(文件过大、边界不清、职责纠缠),把针对性改进纳入设计——就像好的开发者会顺手改善正在改的代码
- 不提无关重构,聚焦服务当前目标

## 设计之后

**设计摘要**:把已验证的设计整理成一份简洁摘要,作为 `/ecc:plan` 的输入。无需长篇文档(spec 的正式落地交给 `/ecc:plan` 或 `/plan-prd`)。

**设计自审**(写完摘要后用新眼光检查):

1. 占位符扫描:有无 "TBD"、"TODO"、空段、模糊需求?修掉
2. 内部一致性:各节是否互相矛盾?架构是否与功能描述匹配?
3. 范围检查:是否聚焦到单个实现计划?需要分解吗?
4. 歧义检查:有无需求可被两种解读?挑一种并明确

就地修复,无需复审。

**用户批准门禁**:自审通过后,请用户确认:

> 设计摘要已整理好。请确认是否同意,我再把它交给 `/ecc:plan` 生成实现计划。

等待回应。要改就改并重新自审;用户批准后才进入终态。

## 终态:交给 /ecc:plan

用户批准后,通过 Skill tool 调用 `/ecc:plan`,把已批准的需求摘要作为参数传入:

```
Skill tool with skill: "ecc:plan", args: "<澄清后、已获批准的需求摘要>"
```

`/ecc:plan` 会重述需求、识别风险、生成分步实现计划,并等待用户确认后才进入实现。

**不要**调用 `tdd-workflow`、`build-fix`、`frontend-design`、`/pr` 或任何其他实现类 skill / agent / command。`brainstorming` 的唯一后继是 `/ecc:plan`。

## 关键原则

- **一次一个问题** —— 别用一堆问题淹没用户
- **多选优先** —— 比开放式更好答
- **YAGNI 无情砍** —— 从所有设计里移除多余功能
- **永远先提 2-3 个方案** —— 再决定,而不是直接拍一个
- **增量验证** —— 呈现一节确认一节
- **保持灵活** —— 不对就回头澄清
