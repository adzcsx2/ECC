# /ecc:brainstorming

动手前的设计探索：把模糊想法通过协作对话逼成清晰、已获批准的设计，再交给 `/ecc:plan` 生成实现计划。设计未经用户批准，绝不进入实现。

---

## 功能

- 新功能开发、新组件、行为修改等任何创意工作开始前使用
- 一次一个问题地澄清需求，避免基于未经验证的假设直接动手
- 提出 2-3 个带权衡的方案，给出推荐及理由
- 分节呈现设计并逐节确认，覆盖架构、组件、数据流、错误处理、测试
- 在既有代码库中遵循既有模式，只做服务于当前目标的针对性改进
- 把已批准的需求摘要交给 `/ecc:plan` 出实现计划

## 用法

```
/ecc:brainstorming <想法或功能的简述>
```

示例：

```
/ecc:brainstorming 给 react todo list 加优先级和筛选
/ecc:brainstorming 为现有查询 API 加一层 Redis 缓存
/ecc:brainstorming 把单体内存存储拆成独立服务
```

## 执行流程

1. **探索项目上下文**：看文件、文档、近期 commit，理解当前结构
2. **评估范围**：若请求涉及多个独立子系统，先标记并分解成子项目，再就第一个子项目走流程
3. **逐个澄清问题**：一次只问一个，搞清目的 / 约束 / 成功标准
4. **提出 2-3 个方案**：带权衡，给出推荐并说明理由
5. **分节呈现设计**：按复杂度伸缩，每节单独确认
6. **写设计摘要并自审**：整理成可喂给 `/ecc:plan` 的简洁摘要，检查占位符 / 矛盾 / 歧义 / 范围
7. **用户批准**：等待明确同意
8. **终态：调用 `/ecc:plan`**：把已批准的需求摘要作为参数传入，由 `/ecc:plan` 重述需求、识别风险、生成分步实现计划

> **硬门禁**：设计获批准前禁止任何实现动作。唯一后继是 `/ecc:plan`。

## 约束

- 设计获用户明确批准前，禁止写代码、调用实现类 skill 或 agent（`tdd-workflow`、`build-fix`、`frontend-design` 等）、搭建脚手架
- 适用于所有项目，包括「看起来很简单」的——简单项目恰恰是未经验证假设造成返工最多的地方
- 一次只问一个问题，多选题优先于开放式提问
- 永远先提出 2-3 个方案再决定，不直接拍一个
- 用 YAGNI 原则从设计中移除多余功能
- 唯一后继是 `/ecc:plan`，不直接实现、不调用其他实现类命令
